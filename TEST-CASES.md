# Verc — Test Cases & Expected Results

End-to-end conversation flows, auto-generated by driving the **real** orchestrator
(NLU + policy engine + Neon DB + NLG + Output Validator). Each case lists the mock input
per turn and the expected result, plus the compliance / math assertions that matter.

> Reproduce any case:
> ```bash
> npm run db:reset
> node --env-file=.env.local --experimental-strip-types scripts/sim.ts <scenario>
> ```
> Scenarios (Part A): `happy · auto · plan_fc · questions · human · bankruptcy · region · dispute · authfail · oneshot · partial`
> Scenarios (Parts B–D): `terse · chatty · typos · injection · pif · bif_installments · lump_haggle · short_plan · decline_fc · cancel_confirm · deceased · military · vod · attorney · prior_paid · cease · credit_question`
>
> Assistant wording is LLM-phrased (NLG) so it varies run to run; the **figures, state
> transitions, and outcomes are deterministic** and are what these cases assert. Settlement
> ladders, plan math, and caps come from the engine — never the LLM.

## Cross-cutting guarantees (all cases)

- **Recording consent → authenticate → disclose** order is enforced by the state machine; no account detail is exposed pre-auth (even "how much do I owe?" only routes to verification).
- **Every disclosed number/date** is injected from account data or computed by the engine — never produced by the LLM. The **Output Validator** blocks any unauthorized figure and logs `OUTPUT_BLOCKED` (unit-tested in `npm test`).
- **Caps are hard ceilings:** the discount ladder ends exactly at the cap and is clamped; plan length is bounded by `effectiveMaxMonths`.
- **Audit log:** every case writes a complete evidence chain (CONSENT_GIVEN, AUTH_*, DISCLOSURE_GIVEN, SETTLEMENT_PROPOSED / PLAN_PROPOSED, FC_*, OFFER_ACCEPTED, ESCALATED, OUTPUT_BLOCKED). View it per session via the UI "View audit log" button or `GET /api/audit?sessionId=…`.
